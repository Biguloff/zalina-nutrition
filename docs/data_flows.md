## PROC-001: Входящий запрос клиента

**Владелец:** Telegram-бот (первичная обработка) → Залина (квалификация)
**Триггер:** Клиент пишет первое сообщение в Telegram-бот
**Частота:** 3-10 раз в неделю
**SLA:** Автоответ — мгновенно; классификация — до 5 минут; уведомление Залине — до 1 минуты после классификации

### Маршрут данных

1. **[КЛИЕНТ]** → Отправляет сообщение в Telegram-бот
   - Вход: текстовое сообщение (свободная форма)
   - Выход: входящее сообщение зарегистрировано
   - Хранилище: `messages` (Telegram API / логи бота)
   - Уведомление: —
   - Цветовой маркер: 🟢 сообщение получено

2. **[БОТ]** → Автоответ + классификация запроса
   - Вход: текст сообщения клиента
   - Выход: тип запроса (`consultation` | `program` | `question` | `other`), приветственное сообщение клиенту
   - Хранилище: `incoming_requests` (тип, timestamp, telegram_id, raw_text)
   - Уведомление: клиенту — приветственное сообщение с описанием услуг
   - Цветовой маркер: 🟢 классифицировано автоматически / 🟡 не удалось классифицировать → передано Залине

3. **[БОТ]** → Отправляет ссылку на анкету
   - Вход: тип запроса, telegram_id клиента
   - Выход: ссылка на Яндекс.Формы (конкретная форма зависит от типа запроса)
   - Хранилище: `incoming_requests.questionnaire_sent_at`
   - Уведомление: клиенту — сообщение со ссылкой на анкету
   - Цветовой маркер: 🟢 ссылка отправлена

4. **[КЛИЕНТ]** → Заполняет анкету (→ переход в PROC-002)
   - Вход: ссылка на форму
   - Выход: заполненная форма
   - Хранилище: Яндекс.Формы (исходные данные)
   - Уведомление: webhook на сервер
   - Цветовой маркер: 🟢 форма заполнена / 🔴 форма не заполнена >48ч

5. **[СЕРВЕР]** → Webhook от Яндекс.Форм: создание записи клиента
   - Вход: JSON-ответ формы (имя, телефон, email, telegram_id)
   - Выход: запись в БД, папка на диске
   - Хранилище: `clients` таблица (id, name, phone, email, telegram_id, status=`new`, created_at)
   - Уведомление: —
   - Цветовой маркер: 🟢 запись создана

6. **[СЕРВЕР]** → Создание папки клиента в Яндекс.Диске
   - Вход: client_id, имя клиента
   - Выход: структура папок: `/{Имя_Фамилия}/01_Анкета/`, `02_Анализы/`, `03_Протоколы/`, `04_Консультации/`, `05_Дневник/`
   - Хранилище: Яндекс.Диск (API)
   - Уведомление: —
   - Цветовой маркер: 🟢 папка создана / 🔴 ошибка API Яндекс.Диска

7. **[СЕРВЕР]** → Уведомление Залине
   - Вход: данные клиента, тип запроса
   - Выход: Telegram-сообщение Залине со ссылкой на дашборд
   - Хранилище: `events` (event_type=`new_client`, client_id, timestamp)
   - Уведомление: Залине в Telegram — "Новый клиент: {Имя}. Тип: {тип}. [Ссылка на дашборд]"
   - Цветовой маркер: 🟢 уведомление доставлено

### Обработка ошибок
- E1: Бот не может классифицировать запрос → пересылка сообщения Залине напрямую + пометка `needs_manual_review`
- E2: Яндекс.Формы webhook не сработал → повторная попытка через 5 мин (до 3 раз) → алерт Залине
- E3: Яндекс.Диск API недоступен → задача в очередь, повтор через 15 мин → ручное создание папки при 3 неудачах
- E4: Клиент не заполнил анкету >48ч → бот отправляет напоминание; >72ч → уведомление Залине

### Стек ИИ/технологий на этом этапе
- **Telegram Bot API**: приём сообщений, отправка ответов — бесплатно
- **Claude Haiku**: классификация типа запроса (consultation/program/question) — ~$0.001/запрос
- **Яндекс.Формы**: хостинг анкеты + webhook — бесплатно (до лимитов)
- **Яндекс.Диск API**: создание папок — бесплатно (в рамках тарифа)

## PROC-002: Заполнение анкеты и квалификация

**Владелец:** Сервер (автоматизация) → Залина (утверждение)
**Триггер:** Webhook от Яндекс.Форм при отправке заполненной анкеты
**Частота:** По мере поступления клиентов (3-10 раз в неделю)
**SLA:** Обработка анкеты — мгновенно; ИИ-анализ — до 30 секунд; уведомление Залине — до 1 минуты

### Маршрут данных

1. **[ЯНДЕКС.ФОРМЫ]** → Webhook с данными анкеты
   - Вход: HTTP POST с JSON-телом (ответы на все вопросы)
   - Выход: сырые данные анкеты на сервере
   - Хранилище: `raw_questionnaire_responses` (JSON blob)
   - Уведомление: —
   - Цветовой маркер: 🟢 webhook получен

2. **[СЕРВЕР]** → Парсинг и нормализация данных
   - Вход: сырой JSON анкеты
   - Выход: структурированный профиль клиента (ФИО, возраст, пол, рост, вес, хронические заболевания, текущие жалобы, цели, аллергии, медикаменты, образ жизни)
   - Хранилище: `clients` (обновление полей), `medical_history` (новая запись: chronic_conditions, allergies, medications, lifestyle)
   - Уведомление: —
   - Цветовой маркер: 🟢 данные распарсены

3. **[ИИ-МОДУЛЬ]** → Предварительный анализ анкеты
   - Вход: структурированный профиль клиента
   - Выход: JSON с флагами: `{risk_flags: [...], contraindications: [...], priority_areas: [...], suggested_tests: [...], confidence: 0.0-1.0}`
   - Хранилище: `ai_assessments` (client_id, type=`questionnaire_pre_analysis`, result_json, confidence, created_at)
   - Уведомление: —
   - Цветовой маркер: 🟢 confidence ≥ 90% / 🟡 confidence 70-89% / 🔴 confidence < 70% или обнаружены серьёзные risk_flags

4. **[СЕРВЕР]** → Сохранение анкеты в файловое хранилище
   - Вход: данные анкеты, client_id
   - Выход: PDF-версия анкеты
   - Хранилище: Яндекс.Диск `/{Клиент}/01_Анкета/анкета_{дата}.pdf`
   - Уведомление: —
   - Цветовой маркер: 🟢 файл сохранён

5. **[СЕРВЕР]** → Уведомление Залине
   - Вход: профиль клиента, результат ИИ-анализа
   - Выход: Telegram-сообщение с кратким резюме
   - Хранилище: `events` (event_type=`questionnaire_completed`, client_id)
   - Уведомление: Залине в Telegram — "Анкета заполнена: {Имя}. Флаги: {risk_flags}. Уверенность ИИ: {confidence}%. [Дашборд]"
   - Цветовой маркер: 🟢 без флагов / 🟡 есть замечания / 🔴 серьёзные противопоказания

6. **[БОТ]** → Подтверждение клиенту
   - Вход: client_id, статус обработки
   - Выход: сообщение клиенту в Telegram
   - Хранилище: `messages_log`
   - Уведомление: клиенту — "Спасибо! Ваша анкета получена и обрабатывается. Залина свяжется с вами в ближайшее время."
   - Цветовой маркер: 🟢 подтверждение отправлено

### Обработка ошибок
- E1: Webhook не доставлен → retry (3 попытки с экспоненциальным backoff) → ручная проверка в Яндекс.Формах
- E2: Парсинг не удался (неожиданная структура формы) → сохранить сырые данные + алерт разработчику
- E3: ИИ-анализ вернул confidence < 50% → пометка `requires_manual_review`, немедленное уведомление Залине
- E4: Обнаружены серьёзные противопоказания (онкология, беременность, приём антикоагулянтов) → немедленный алерт Залине с пометкой URGENT

### Стек ИИ/технологий на этом этапе
- **Claude Sonnet**: предварительный анализ анкеты, выявление рисков и противопоказаний — ~$0.01/анкета
- **Яндекс.Формы Webhook**: доставка данных — бесплатно
- **ReportLab**: генерация PDF-версии анкеты — бесплатно (open source)

## PROC-003: Оплата

**Владелец:** Telegram-бот (отправка ссылки) → Tinkoff API (процессинг) → Сервер (учёт)
**Триггер:** Залина подтверждает готовность к работе с клиентом / клиент выбирает тариф
**Частота:** По мере квалификации клиентов
**SLA:** Ссылка на оплату — мгновенно; webhook от Tinkoff — до 30 секунд; напоминания по расписанию

### Маршрут данных

1. **[БОТ]** → Отправляет варианты тарифов (inline-кнопки)
   - Вход: client_id, доступные тарифы из конфигурации
   - Выход: сообщение с кнопками (например: "Разовая консультация — 5 000 ₽", "Интенсив 30 дней — 25 000 ₽")
   - Хранилище: `messages_log`
   - Уведомление: клиенту — сообщение с выбором тарифа
   - Цветовой маркер: 🟢 тарифы отправлены

2. **[КЛИЕНТ]** → Выбирает тариф (нажимает кнопку)
   - Вход: callback_query от Telegram
   - Выход: выбранный тариф (tariff_id, amount, description)
   - Хранилище: `payments` (client_id, tariff_id, amount, status=`pending`, created_at)
   - Уведомление: —
   - Цветовой маркер: 🟢 тариф выбран

3. **[СЕРВЕР]** → Генерация платёжной ссылки через Tinkoff API
   - Вход: amount, order_id (= payment_id), description, client email
   - Выход: payment_url (одноразовая ссылка на оплату)
   - Хранилище: `payments.payment_url`, `payments.tinkoff_payment_id`
   - Уведомление: —
   - Цветовой маркер: 🟢 ссылка сгенерирована / 🔴 ошибка Tinkoff API

4. **[БОТ]** → Отправляет ссылку на оплату клиенту
   - Вход: payment_url, описание тарифа
   - Выход: сообщение с кнопкой "Оплатить"
   - Хранилище: `messages_log`
   - Уведомление: клиенту — "Ссылка на оплату: {тариф} — {сумма} ₽. [Оплатить]"
   - Цветовой маркер: 🟢 ссылка отправлена

5. **[КЛИЕНТ]** → Совершает оплату
   - Вход: платёжная страница Tinkoff
   - Выход: результат транзакции (успех/неудача)
   - Хранилище: Tinkoff (внутренний учёт)
   - Уведомление: —
   - Цветовой маркер: 🟢 оплата прошла / 🔴 оплата отклонена

6. **[TINKOFF]** → Webhook подтверждения оплаты
   - Вход: HTTP POST (payment_id, status, amount, timestamp)
   - Выход: подтверждение серверу
   - Хранилище: `payments` (status → `paid`, paid_at, tinkoff_response_json)
   - Уведомление: —
   - Цветовой маркер: 🟢 webhook получен, статус обновлён

7. **[СЕРВЕР]** → Обновление статуса клиента + уведомления
   - Вход: payment_id, статус оплаты
   - Выход: обновление клиента, рассылка уведомлений
   - Хранилище: `clients.status` → `paid`, `events` (event_type=`payment_received`)
   - Уведомление: клиенту — "Оплата получена! Спасибо. Мы начинаем работу." / Залине — "Оплата от {Имя}: {сумма} ₽. Тариф: {тариф}. [Дашборд]"
   - Цветовой маркер: 🟢 все уведомления доставлены

8. **[БОТ]** → Напоминания при неоплате (каскад)
   - Вход: cron-проверка payments где status=`pending`
   - Выход: напоминания по расписанию
   - Хранилище: `payment_reminders` (payment_id, reminder_number, sent_at)
   - Уведомление:
     - 24ч без оплаты → клиенту: "Напоминаем об оплате. Ссылка актуальна 72 часа."
     - 48ч без оплаты → клиенту: "Ссылка на оплату истекает через 24 часа."
     - 72ч без оплаты → Залине: "Клиент {Имя} не оплатил в течение 72ч. Требуется решение."
   - Цветовой маркер: 🟡 24ч / 🟡 48ч / 🔴 72ч

### Обработка ошибок
- E1: Tinkoff API недоступен → retry через 2 мин (до 5 раз) → уведомление Залине, ручная отправка реквизитов
- E2: Webhook от Tinkoff не пришёл → проверка статуса через polling каждые 5 мин в течение часа
- E3: Оплата отклонена банком → сообщение клиенту с просьбой повторить или использовать другую карту
- E4: Сумма webhook не совпадает с ожидаемой → алерт Залине + блокировка автоматического подтверждения

### Стек ИИ/технологий на этом этапе
- **Tinkoff Acquiring API**: генерация ссылок, приём оплат — комиссия ~2.5%
- **Telegram Bot API (inline keyboards)**: интерфейс выбора тарифа — бесплатно
- **Cron-задача**: проверка неоплаченных счетов — бесплатно

## PROC-004: Сдача и разбор анализов крови

**Владелец:** Клиент (предоставление) → ИИ (черновой разбор) → Залина (финальное утверждение)
**Триггер:** Клиент отправляет фото/PDF анализов в бот
**Частота:** 1-2 раза за цикл работы с клиентом
**SLA:** OCR + ИИ-анализ — до 2 минут; разбор Залиной — до 24 часов; отправка клиенту — до 1 часа после утверждения

### Маршрут данных

1. **[КЛИЕНТ]** → Отправляет фото или PDF анализов в бот
   - Вход: изображение (JPG/PNG) или PDF-файл
   - Выход: файл принят ботом
   - Хранилище: временное хранилище Telegram (file_id)
   - Уведомление: клиенту — "Анализы получены! Обрабатываю..."
   - Цветовой маркер: 🟢 файл получен

2. **[БОТ]** → Сохранение файла на Яндекс.Диск
   - Вход: file_id из Telegram
   - Выход: файл на диске
   - Хранилище: Яндекс.Диск `/{Клиент}/02_Анализы/анализы_{дата}.{ext}`
   - Уведомление: —
   - Цветовой маркер: 🟢 файл сохранён / 🔴 ошибка загрузки

3. **[ИИ-МОДУЛЬ]** → OCR: извлечение значений из анализов
   - Вход: изображение/PDF анализов
   - Выход: структурированная таблица: `[{marker_name, value, unit, reference_range}, ...]`
   - Хранилище: `ocr_results` (client_id, raw_extracted_json, confidence, created_at)
   - Уведомление: —
   - Цветовой маркер: 🟢 confidence ≥ 90% / 🟡 70-89% / 🔴 < 70%

4. **[ИИ-МОДУЛЬ]** → Сравнение с референсными диапазонами
   - Вход: извлечённые значения, таблица референсных диапазонов (с учётом пола, возраста)
   - Выход: классификация каждого маркера: `{marker, value, ref_range, status: green|amber|red, deviation_pct}`
   - Хранилище: `lab_markers` (client_id, lab_result_id, marker_name, value, ref_min, ref_max, status, deviation)
   - Уведомление: —
   - Цветовой маркер: 🟢 в норме / 🟡 пограничное значение (отклонение 10-25%) / 🔴 вне нормы (отклонение > 25%)

5. **[ИИ-МОДУЛЬ]** → Генерация чернового анализа
   - Вход: классифицированные маркеры, профиль клиента (из medical_history), анамнез
   - Выход: текстовый черновик анализа: общая картина, проблемные зоны, взаимосвязи маркеров, предварительные рекомендации
   - Хранилище: `lab_results` (client_id, draft_analysis_text, ai_confidence, status=`draft`, created_at)
   - Уведомление: —
   - Цветовой маркер: 🟡 черновик создан, ожидает разбора

6. **[ДАШБОРД]** → Отображение статуса "Ожидает разбора"
   - Вход: lab_result с status=`draft`
   - Выход: карточка в дашборде с индикатором
   - Хранилище: —
   - Уведомление: Залине в Telegram — "Новые анализы: {Имя}. 🔴 {N} маркеров вне нормы, 🟡 {M} пограничных. [Открыть в дашборде]"
   - Цветовой маркер: 🟡 ожидает разбора

7. **[ЗАЛИНА]** → Ревью ИИ-черновика в дашборде
   - Вход: черновик анализа, исходные данные, профиль клиента
   - Выход: утверждённый/отредактированный анализ
   - Хранилище: `lab_results` (status → `approved`, final_analysis_text, reviewed_by=`zalina`, reviewed_at)
   - Уведомление: —
   - Цветовой маркер: 🟢 утверждён / 🟡 утверждён с правками

8. **[СЕРВЕР]** → Генерация PDF-рекомендаций (ReportLab)
   - Вход: утверждённый анализ, данные маркеров, профиль клиента
   - Выход: стилизованный PDF (формат A4, дизайн "Warm Organic": кремовый фон, шалфейные и медные акценты)
   - Хранилище: Яндекс.Диск `/{Клиент}/03_Протоколы/анализ_крови_{дата}.pdf`
   - Уведомление: —
   - Цветовой маркер: 🟢 PDF сгенерирован

9. **[БОТ]** → Отправка PDF клиенту
   - Вход: PDF-файл, client_id
   - Выход: сообщение с документом в Telegram
   - Хранилище: `events` (event_type=`lab_results_sent`, client_id, timestamp)

   - Цветовой маркер: 🟢 улучшение / 🟡 стабильно / 🔴 ухудшение

7. **[СЕРВЕР]** → Генерация финального отчёта с визуализацией дельт
   - Вход: сравнение анализов, все measurements за программу, food_diary статистика, daily_tracking статистика, финальный опрос
   - Выход: комплексный PDF-отчёт (до/после по анализам, графики прогресса замеров, статистика дневника, NPS)
   - Хранилище: Яндекс.Диск `/{Клиент}/03_Протоколы/финальный_отчёт_{дата}.pdf`
   - Уведомление: —
   - Цветовой маркер: 🟢 отчёт сгенерирован

8. **[ЗАЛИНА]** → Финальная Zoom-консультация (→ PROC-005)
   - Вход: финальный отчёт, сравнение анализов
   - Выход: запись, заметки (см. PROC-005), персональные рекомендации на будущее
   - Хранилище: см. PROC-005
   - Уведомление: —
   - Цветовой маркер: 🟢 финальная консультация проведена

9. **[БОТ]** → Предложение продления (inline-кнопки)
   - Вход: client_id, доступные варианты
   - Выход: сообщение с кнопками выбора
   - Хранилище: `messages_log`
   - Уведомление: клиенту — "Спасибо за прохождение программы! Хотите продолжить?"
     - [Продолжить 1 месяц]
     - [Продолжить 2 месяца]
     - [Продолжить 3 месяца]
     - [Завершить]
   - Цветовой маркер: 🟢 предложение отправлено

10. **[КЛИЕНТ]** → Принимает решение
    - Вход: нажатие кнопки
    - Выход: решение клиента
    - Хранилище: `program_renewals` (client_id, decision, selected_duration, created_at)
    - Уведомление: Залине — "{Имя} решил(а): {решение}"
    - Маршрутизация:
      - **Продолжить** → генерация платёжной ссылки (→ PROC-003) → после оплаты → новый цикл (→ PROC-007)
      - **Завершить** → шаг 11
    - Цветовой маркер: 🟢 продление / 🔴 завершение

11. **[СЕРВЕР]** → Завершение и архивация (при отказе от продления)
    - Вход: client_id, решение=`decline`
    - Выход: архивация данных
    - Хранилище:
      - `clients.status` → `completed`
      - `program_status.status` → `completed`
      - Яндекс.Диск: папка клиента → перемещение в `/_Архив/{Год}/{Имя}/`
    - Уведомление: клиенту — "Спасибо за доверие! Все ваши материалы сохранены. Мы всегда на связи."
    - Цветовой маркер: 🟢 архивировано

12. **[CRON]** → Ретеншн-напоминания (каскад)
    - Вход: clients со status=`completed`, дата завершения
    - Выход: серия напоминаний по расписанию
    - Хранилище: `retention_reminders` (client_id, reminder_type, sent_at)
    - Уведомление:
      - **30 дней после завершения**: "Как ваше самочувствие? Если хотите продолжить работу — мы здесь. [Записаться на консультацию]"
      - **90 дней**: "Прошло 3 месяца. Рекомендуем сдать контрольные анализы. [Список анализов]"
      - **6 месяцев**: "Полгода! Время проверить, как закрепились привычки. [Записаться]"
    - Цветовой маркер: 🟡 напоминание отправлено / 🟢 клиент вернулся / 🔴 нет реакции на все напоминания

### Обработка ошибок
- E1: Клиент не заполнил финальный опрос → напоминание через 24ч и 48ч → Залина напоминает на финальном Zoom
- E2: Клиент не сдал повторные анализы → финальный отчёт генерируется без сравнения, с пометкой "Повторные анализы не сданы"
- E3: NPS = Detractor (0-6) → немедленное уведомление Залине с пометкой URGENT для личного follow-up
- E4: Архивация на Яндекс.Диске не удалась → повторная попытка, алерт разработчику
- E5: Клиент не реагирует на ретеншн-напоминания → после 6 месяцев статус → `inactive`, прекращение рассылки

### Стек ИИ/технологий на этом этапе

- **Tinkoff API**: платёж при продлении — комиссия ~2.5%
- **Cron**: ретеншн-напоминания — бесплатно

# Сквозные процессы

## Трекинг вовлечённости (cross-cutting)

**Владелец:** Сервер (автоматический расчёт) → Дашборд (визуализация) → Залина (контроль)
**Триггер:** Каждое взаимодействие клиента с ботом
**Частота:** Непрерывно (расчёт score — раз в час; дайджест — ежедневно; сводка — еженедельно)

### Маршрут данных

1. **[СЕРВЕР]** → Регистрация каждого взаимодействия
   - Вход: любое действие клиента (сообщение, нажатие кнопки, отправка фото, ввод данных)
   - Выход: событие в логе
   - Хранилище: `engagement_events` (client_id, event_type, timestamp, weight)
   - Весовые коэффициенты:
     - Отправка фото еды: 3 балла
     - Ввод веса: 2 балла
     - Заполнение вечернего чек-листа: 2 балла
     - Ответ на сообщение бота: 1 балл
     - Бронирование Zoom: 3 балла

2. **[СЕРВЕР]** → Расчёт ежедневного engagement score (0-100%)
   - Вход: все engagement_events за день, максимально возможный score за день
   - Выход: daily_score = (набранные баллы / максимум) * 100%
   - Хранилище: `engagement_scores` (client_id, date, daily_score, events_count)
   - Уведомление: —
   - Цветовой маркер: 🟢 ≥ 80% / 🟡 40-79% / 🔴 < 40%

3. **[ДАШБОРД]** → Отображение вовлечённости по каждому клиенту
   - Вход: engagement_scores, тренд за последние 7 дней
   - Выход: виджет: полоса прогресса + тренд (стрелка вверх/вниз/ровно)
   - Хранилище: дашборд (кэш)
   - Уведомление: —
   - Цветовой маркер: цвет полосы соответствует score

4. **[CRON 21:00 ежедневно]** → Ежедневный дайджест Залине
   - Вход: все engagement_scores за день, все food_diary с 🟡, пропущенные задания
   - Выход: сводное Telegram-сообщение
   - Хранилище: `digests` (date, content_json, sent_at)
   - Уведомление: Залине — "Дайджест за {дата}: {N} клиентов активны. Отклонения: {список}. Пропущенные задания: {список}."

5. **[CRON понедельник 09:00]** → Еженедельная сводка Залине
   - Вход: engagement_scores за неделю по всем клиентам
   - Выход: рейтинг и алерты
   - Хранилище: `weekly_digests`
   - Уведомление: Залине — "Еженедельная сводка: Топ-3 активных: {имена, scores}. Теряют мотивацию: {имена, scores, тренд}. Общий engagement: {средний}%."

6. **[СЕРВЕР]** → Авто-эскалация при падении вовлечённости
   - Вход: engagement_scores
   - Выход: немедленное уведомление
   - Условие: daily_score < 40% три дня подряд
   - Хранилище: `escalations` (client_id, reason, created_at)
   - Уведомление: Залине — "🔴 {Имя}: вовлечённость ниже 40% уже {N} дней подряд. Рекомендуется личное сообщение. [Дашборд]"
   - Цветовой маркер: 🔴 эскалация

### Обработка ошибок
- E1: Данные для расчёта score не полные (например, бот не работал) → использовать последний известный score, пометить как `estimated`
- E2: Все клиенты показывают 0% (сбой системы) → алерт разработчику, не отправлять дайджест с нулями

## Система уведомлений Залины (notification routing)

**Владелец:** Маршрутизатор уведомлений (серверный компонент)
**Триггер:** Любое событие, требующее внимания Залины
**Частота:** Непрерывно

### Уровни приоритетов

#### ВЫСОКИЙ (немедленный Telegram)

| Событие | Источник | Формат уведомления |
|---------|----------|-------------------|
| Нежелательные реакции | Сообщение клиента (ключевые слова: "плохо", "тошнит", "аллергия", "сыпь") | "🔴 СРОЧНО: {Имя} сообщает о нежелательной реакции: '{текст}'. [Открыть чат]" |
| Анализы крови сданы | PROC-004 шаг 1 | "🩸 {Имя} отправил(а) анализы крови. [Открыть разбор]" |
| Нет ответа 3+ дня | Engagement tracker | "🔴 {Имя}: нет активности {N} дней. Рекомендуется связаться лично." |
| Оплата получена | PROC-003 шаг 7 | "💰 Оплата от {Имя}: {сумма} ₽. Тариф: {тариф}. [Дашборд]" |
| NPS Detractor | PROC-012 шаг 3 | "🔴 {Имя} — Detractor (NPS {score}). Требуется follow-up." |

#### СРЕДНИЙ (ежедневный дайджест в 21:00)

| Событие | Источник | Формат в дайджесте |
|---------|----------|-------------------|
| Отклонения в питании >30% | PROC-009 (🟡 рейтинги) | "{Имя}: {N} приёмов пищи с замечаниями. Основное: {рекомендации}" |
| Пропущено 2+ приёма пищи | food_diary (нет записей) | "{Имя}: пропущено {N} приёмов пищи за день" |
| Вопросы клиентов | Сообщения в боте | "{Имя}: '{вопрос}'. [Ответить]" |
| Неоплаченный счёт 24ч | PROC-003 шаг 8 | "{Имя}: счёт не оплачен {часов}ч" |

#### НИЗКИЙ (еженедельная сводка, понедельник 09:00)

| Событие | Источник | Формат в сводке |
|---------|----------|-----------------|
| Незначительные отклонения | daily_tracking (score 60-79%) | "Клиенты с умеренным engagement: {список}" |
| Метрики в норме | measurements, daily_tracking | "На графике: {N} клиентов — положительная динамика" |
| Статистика engagement | engagement_scores | "Средний engagement: {N}%. Тренд: {стрелка}" |
| Статистика NPS | final_surveys | "NPS за месяц: Promoters {N}, Passives {N}, Detractors {N}" |

### Маршрут данных

1. **[СОБЫТИЕ]** → Регистрация в очереди уведомлений
   - Вход: event_type, client_id, data, priority (auto-determined)
   - Хранилище: `notification_queue` (id, event_type, client_id, priority, payload_json, status=`pending`, created_at)

2. **[МАРШРУТИЗАТОР]** → Обработка по приоритету
   - HIGH → немедленная отправка через Telegram Bot API
   - MEDIUM → добавление в `daily_digest_buffer`
   - LOW → добавление в `weekly_digest_buffer`

3. **[CRON 21:00]** → Сборка и отправка ежедневного дайджеста
   - Вход: все записи из `daily_digest_buffer` за день
   - Выход: одно форматированное Telegram-сообщение
   - Хранилище: `digests`, `notification_queue` (status → `sent`)

4. **[CRON понедельник 09:00]** → Сборка и отправка еженедельной сводки
   - Вход: все записи из `weekly_digest_buffer` за неделю
   - Выход: одно форматированное Telegram-сообщение с секциями
   - Хранилище: `weekly_digests`, `notification_queue` (status → `sent`)

### Обработка ошибок
- E1: Telegram API недоступен для HIGH-priority → retry каждую минуту (до 10 раз) → fallback: SMS / email
- E2: Дайджест пустой (нет событий) → не отправлять, логировать
- E3: Дубликаты уведомлений → дедупликация по event_type + client_id + 5-минутное окно

## ИИ-шлюз одобрения (AI approval gate)

**Владелец:** Сервер (автоматическая маршрутизация) → Залина (финальное одобрение для критических решений)
**Триггер:** Любой ИИ-результат, требующий решения о публикации клиенту
**Частота:** Непрерывно (каждый ИИ-запрос проходит через шлюз)

### Уровни уверенности и действия

| Уровень уверенности ИИ | Действие | Визуальный маркер | Уведомление Залине |
|-------------------------|----------|-------------------|-------------------|
| **≥ 90%** | Авто-одобрение + логирование для пакетной проверки | 🟢 в дашборде | Нет (включается в еженедельный batch review) |
| **70-89%** | Авто-обработка + флаг для выборочной проверки | 🟡 в дашборде | Нет (помечается для spot-check) |
| **< 70%** | Удержание для ручной проверки | 🔴 в дашборде | Немедленное Telegram: "ИИ не уверен ({confidence}%). Требуется ручная проверка. [Дашборд]" |

### ВСЕГДА требуется одобрение Залины (независимо от confidence):

| Категория | Причина | Процесс |
|-----------|---------|---------|
| Интерпретация анализов крови | Медицинская ответственность | ИИ создаёт черновик → Залина ревьюит и утверждает → только потом отправка клиенту |
| Изменения протокола нутрицевтиков | Влияние на здоровье | Любые предложения по изменению → утверждение Залиной |
| Корректировки дозировок | Безопасность | ИИ может предложить → Залина решает |
| Любые медицинские рекомендации | Регуляторные требования | Всегда ручное утверждение |

### Маршрут данных

1. **[ИИ-МОДУЛЬ]** → Генерация результата с confidence score
   - Вход: запрос (анализ фото, разбор анализов, генерация рекомендаций)
   - Выход: результат + confidence (0.0-1.0) + категория (food_analysis | lab_interpretation | protocol_change | general)
   - Хранилище: `ai_results` (id, type, input_hash, output_json, confidence, model, tokens_used, cost, created_at)

2. **[ШЛЮЗ]** → Определение маршрута
   - Вход: ai_result, категория, confidence
   - Выход: решение о маршруте (auto_approve | flag_for_review | hold_for_manual)
   - Логика:
     ```
     if category in [lab_interpretation, protocol_change, dosage, medical]:
         route = HOLD_FOR_MANUAL  # ВСЕГДА ручная проверка
     elif confidence >= 0.90:
         route = AUTO_APPROVE
     elif confidence >= 0.70:
         route = FLAG_FOR_REVIEW
     else:
         route = HOLD_FOR_MANUAL
     ```
   - Хранилище: `ai_results` (route, approved_at, approved_by)

3. **[AUTO_APPROVE]** → Автоматическая публикация
   - Вход: ai_result с route=AUTO_APPROVE
   - Выход: результат передаётся в соответствующий процесс (бот отправляет клиенту)
   - Хранилище: `ai_results` (approved_by=`auto`, approved_at=now())
   - Уведомление: нет (попадает в batch review раз в неделю)
   - Цветовой маркер: 🟢

4. **[FLAG_FOR_REVIEW]** → Авто-обработка с пометкой
   - Вход: ai_result с route=FLAG_FOR_REVIEW
   - Выход: результат передаётся клиенту + флаг в дашборде
   - Хранилище: `ai_results` (approved_by=`auto_flagged`, needs_review=true)
   - Уведомление: нет (помечено 🟡 в дашборде для spot-check)
   - Цветовой маркер: 🟡

5. **[HOLD_FOR_MANUAL]** → Удержание до ручной проверки
   - Вход: ai_result с route=HOLD_FOR_MANUAL
   - Выход: результат НЕ передаётся клиенту до одобрения
   - Хранилище: `ai_results` (status=`pending_review`)
   - Уведомление: Залине — немедленный Telegram: "🔴 Требуется ручная проверка: {тип}. Клиент: {Имя}. Уверенность ИИ: {confidence}%. [Открыть в дашборде]"
   - Цветовой маркер: 🔴

6. **[ЗАЛИНА]** → Ручная проверка и решение
   - Вход: ai_result в дашборде
   - Выход: approve / reject / edit
   - Хранилище: `ai_results` (approved_by=`zalina`, approved_at, review_notes)
   - Уведомление: при approve → результат отправляется клиенту; при reject → логирование причины, результат удаляется
   - Цветовой маркер: 🟢 одобрено / 🔴 отклонено

7. **[CRON понедельник]** → Пакетная проверка авто-одобренных результатов
   - Вход: все ai_results с approved_by=`auto` за неделю
   - Выход: список для проверки Залиной в дашборде
   - Хранилище: `batch_reviews` (week, items_count, reviewed_count)
   - Уведомление: Залине — "Пакетная проверка: {N} авто-одобренных ИИ-результатов за неделю. [Открыть]"

### Обработка ошибок
- E1: ИИ не вернул confidence score → route = HOLD_FOR_MANUAL по умолчанию
- E2: Залина не проверила HOLD_FOR_MANUAL >4ч → повторное уведомление с повышением приоритета
- E3: ИИ-модель недоступна → все запросы в очередь, уведомление разработчику, Залине — "ИИ-анализ временно недоступен"
- E4: Расхождение между авто-одобренным результатом и ручной проверкой (на batch review) → пересчёт порогов, логирование для улучшения модели

### Стек ИИ/технологий на этом этапе
- **GPT-4o Vision**: анализ фото — ~$0.03/запрос
- **Claude Sonnet**: анализ текста, генерация рекомендаций — ~$0.01-0.03/запрос
- **Claude Haiku**: классификация, простые задачи — ~$0.001/запрос
- **PostgreSQL**: хранение ai_results, маршрутизация — бесплатно (self-hosted)
- **Redis**: очередь задач, кэш — бесплатно (self-hosted)
